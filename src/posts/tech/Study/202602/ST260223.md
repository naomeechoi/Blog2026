## C++ 중요 내용 복습
1) 레퍼런스와 포인터의 차이
레퍼런스는 변수의 또다른 이름, 별칭이다. 선언과 동시에 초기화 해야하고, 한번 바인딩되면 다른 객체로 바꿀 수 없다. null을 허용하지 않는다. 즉 레퍼런스는 항상 유효한 객체를 참조하는 안전한 별칭이다. 레퍼런스의 크기는 보통 포인터처럼 저장되기 때문에 64비트 환경에서 8바이트 크기를 갖는다. 그러나 최적화 후에는 컴파일러가 래퍼런스를 위한 별도 공간을 만들지 않을 수도 있다. 포인터는 메모리 주소를 저장하는 변수이다. null을 가질 수 있고, 다른 객체를 가리키도록 변경할 수 있으며 주소 연산이 가능하다.
</br></br>
2) 스마트 포인터
스마트 포인터는 동적 할당된 객체의 생명주기를 자동으로 관리하기 위한 도구이다. 객체가 스코프를 벗어나면 자동으로 delete가 호출되도록 설계되어 있다. unique_ptr은 단독 소유권을 가지며 복사는 불가능하고 이동만 가능하다. shared_ptr은 참조 카운트를 기반으로 여러 객체가 공동 소유할 수 있다. weak_ptr은 shared_ptr이 소유한 객체를 참조하되 소유권에는 관여하지 않는 약한 참조이다.
shared_ptr 끼리 서로 참조할 경우 순환 참조가 발생할 수 있다. 서로 참조하면서 참조 카운트가 0이 되지 않아 삭제가 되지 않고 메모리 누수가 생긴다. 이 때 한 쪽을 weak_ptr로 바꾸면 참조 카운트를 증가시키지 않는다. 즉, 객체 소유권을 가지지 않고도 접근할 수 있다. weak_ptr 자체는 객체가 살아있는지 확인할 수 없고 소유권도 없다. lock()을 사용하면 살아있는 객체에 대해 임시 shared_ptr을 만들어 안전하게 접근 가능하다. 객체가 살아있으면 참조 카운트가 증가하고 소멸되었으면 nullptr이 반환된다. 
</br></br>
3) 연산자 오버로딩
연산자 오버로딩은 기존 연산자 기호의 의미를 재정의하는 기능이다. 최소 한 개의 피연산자는 사용자 정의 타입이여야 한다. 기본 타입끼리의 연산은 이미 언어가 정의하고 있기 때문에, 오버로딩하는 것이 허용되지 않는다.
</br></br>
4) 함수 내 static keyword의 역할
함수 내부에 선언된 static 변수는 일반 지역 변수와 달리 데이터 영역에 할당된다. 따라서 함수가 여러번 호출되더라도 초기화는 한번 수행되고 값은 유지 된다. 함수 종료 후에도 값이 사라지지 않고 다음 호출 시 그대로 사용할 수 있다. 함수가 속성을 지니게 할 수 있다.
</br></br>
5) RTTI 구현해서 쓰는 이유
RTTI는 C++에서 런타임 객체의 실제 타입을 확인할 수 있게 해준다. 부모클래스 포인터가 가리키는 객체가 실제로 어떤 파생 클래스인지 확인하거나 안전하게 다운캐스팅을 수행할 때 필요하다. RTTI를 직접 구현해서 쓰는 이유는 필요한 정보만 최소화하여 테이블을 경량화하거나 캐싱하여 사용할 수 있다. 런타임 타입 테이블 조회에 대한 비용을 줄일 수 있다. 또한 RTTI를 지원하지 않는 환경에서 사용 가능하다. 표준 RTTI가 제공하지 않는 런타임 타입 기반 로직을 구현 가능하다.(예를 들어 파생 클래스마다 ID 부여하여 처리)
</br></br>
6) 해시맵
만약 어떤 값들을 배열 같은 컨테이너에 순차적으로 저장한다면, 원하는 값을 찾기 위해 O(N) 순차 탐색을 해야한다.
이를 개선하기 위해 사용하는 자료구조가 해시맵이다. 해시맵은 키를 해시함수를 통해 숫자로 변환하고 그 값을 인덱스로 사용하여 버킷에 값을 저장한다.
따라서 평균 시간 복잡도가 O(1)이지만, 해시 값을 계산하는 해시 함수의 성능이 중요하다.
해시 함수는 계산이 빨라야하고, 충돌이 적어야 하며, 키를 균등하게 분산시켜야 한다.
서로 다른 키가 같은 해시값을 갖는 현상을 해시 충돌이라고 한다. 해결하기 위한 방법으로 오픈어드레싱, 체이닝 방법이 있다.
오픈어드레싱은 해시 충돌이 일어나면 다른 빈 버킷을 탐색하여 저장하는 방식이다. 모든 데이터가 배열 내부에 저장되고, 메모리가 연속적이라 캐시 친화적이다.
중간 데이터를 삭제하면 탐색 경로가 끊길 수 있기 때문에 실제 삭제 대신 더미를 남겨둔다.
해시테이블에 빈 공간이 적어질 수록, 충돌이 발생했을 때 O(N)의 속도로 빈칸을 찾게 된다. 충돌 때문에 데이터가 한 구간에 몰리는 클러스터링 현상이 생겨서 특정 구간의 해시값들에서 속도가 느려지는 현상이 발생한다. 따라서 리사이징이 필요하다.
체이닝은 보통 링크드리스트를 두어 같은 해시값을 갖는 데이터를 연결한다. 구현이 단순하고 버킷에 빈 공간이 적어져도 영향이 상대적으로 적다. 그러나 노드가 메모리에 흩어져 있어 캐시 효율이 낮고, 해시 충돌이 높아지면 O(N)의 속도에 가까워진다. 이를 해결하기 위해 체이닝의 크기가 커질 경우 트리를 사용한다.
해시 중복을 해결하기 위해서는 버킷의 사이즈가 커야한다. 또한 버킷의 숫자를 소수로 할 경우에 분산이 더 균등해진다. 하지만 현대에는 2의 거듭제곱 크기를 버킷의 숫자로 이용하여 % 연산보다 빠른 "& (M - 1)" 형식의 비트 연산을 사용한다.
</br></br>
7) 전방선언 사용 이유
포인터를 선언해 멤버 변수로 사용할 경우, 헤더에서 해당 타입의 크기나 내부 구조를 알 필요 없기 떄문에 완전한 타입 정의가 필요하지 않다. 따라서 이름만 선언하는 전방선언을 사하면 헤더에 인클루드 되는 코드를 줄일 수 있다. 두 개 이상의 헤더 파일이 서로를 포함하면서 무한 의존 관계가 생기는 순환 참조를 해결할 수 있다. 서로가 서로를 필요로 하면서 완전한 타입 정보를 얻지 못한다.
